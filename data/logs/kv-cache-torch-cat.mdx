KV caching is fundamental to efficient LLM inference. If you've implemented autoregressive decoding in PyTorch, you might have written something like this:

```python
if self.k_cache is None:
    self.k_cache = K
    self.v_cache = V
else:
    self.k_cache = torch.cat([self.k_cache, K], dim=2)
    self.v_cache = torch.cat([self.v_cache, V], dim=2)
```

This pattern makes logical sense. Search "KV cache PyTorch implementation" and you'll find it in [Sebastian Raschka's excellent tutorial on KV caching](https://magazine.sebastianraschka.com/p/coding-the-kv-cache-in-llms), [HuggingFace's KV caching blog post](https://huggingface.co/blog/not-lain/kv-caching), and countless GitHub repositories. It's pedagogically clear: at each decode step, concatenate the new key-value pair onto the existing cache.

But this pattern has a hidden performance trap.

> [!info] Who is this for?
>
> This post is for ML practitioners who implement or optimize LLM inference. If you've written a KV cache using `torch.cat` and are deploying to production, this is especially relevant. If you're learning about attention mechanisms, the tutorial implementations are fine—just be aware of this limitation when scaling up.

> [!tip] tl;dr
>
> `torch.cat` copies the entire existing cache at every decode step, resulting in $O(N^2)$ total memory operations over $N$ steps. The fix is simple: pre-allocate a buffer and use slice assignment. Jump to [The Fix](#the-fix-pre-allocated-buffers-with-slice-assignment) if you just want the solution.

# The Hidden Quadratic Cost of torch.cat

Let's trace through what actually happens when you call `torch.cat([self.k_cache, K], dim=2)` at decode step $t$:

1. **Allocate** a new tensor of size $t + 1$ (the combined length)
2. **Copy** all $t$ existing cache elements to the new tensor
3. **Copy** the 1 new element to the end
4. **Deallocate** the old tensor (eventually, via garbage collection)

The critical observation: **every single decode step copies the entire existing cache**. At step $t$, we perform $O(t)$ memory copy operations.

The cumulative cost over $N$ decode steps is:

$$\text{Total memory operations} = \sum_{t=1}^{N} t = \frac{N(N+1)}{2} = O(N^2)$$

This is **in addition to** the unavoidable $O(N^2)$ cost of the attention computation itself. We've introduced a second quadratic term, purely from memory allocation patterns.

> [!warning] The Hidden Cost
>
> Every `torch.cat` doesn't just add one element—it copies the entire existing cache to a new memory location. At step 1000, you're copying 999 elements just to add 1.

To make this concrete, here's what `torch.cat` is doing under the hood:

```python
def torch_cat_equivalent(existing_cache, new_tensor, dim):
    # Step 1: Allocate NEW memory for combined size
    new_size = existing_cache.size(dim) + new_tensor.size(dim)
    new_cache = torch.empty(..., new_size, ...)  # O(N) allocation

    # Step 2: Copy ALL existing data to new location
    new_cache[..., :existing_cache.size(dim)] = existing_cache  # O(N) copy!

    # Step 3: Copy new data
    new_cache[..., existing_cache.size(dim):] = new_tensor  # O(1) copy

    return new_cache
    # Old cache becomes garbage — memory pressure until GC runs
```

# Memory Access Patterns: Why This Hurts

GPU memory bandwidth is precious. Modern GPUs like the H100 have enormous compute throughput (around 2 petaFLOPS for FP16), but memory bandwidth is comparatively limited (~3 TB/s for HBM3). Moving data is often the bottleneck, not computing on it.

> [!info] Memory-Bound Operations
>
> An operation is "memory-bound" when its runtime is dominated by data movement rather than arithmetic. During autoregressive decoding, we process just one token at a time, meaning compute utilization is low and memory access patterns dominate latency.

The `torch.cat` pattern fails on multiple fronts:

- **Allocation overhead**: `cudaMalloc` is not free. Each allocation involves driver overhead and potentially memory pool management.
- **Full cache copy every step**: Reading and writing $O(t)$ elements at step $t$ consumes bandwidth that could be used for actual computation.
- **Memory fragmentation**: Repeatedly allocating slightly larger buffers fragments the memory pool, potentially forcing expensive compaction or causing allocation failures.
- **Cache unfriendliness**: The cache moves to new memory addresses each step, defeating any benefits from hardware caching or memory prefetching.

# The Fix: Pre-allocated Buffers with Slice Assignment

The solution is straightforward: allocate the full buffer upfront and write to it in place.

```python
# Instead of this (quadratic):
self.k_cache = torch.cat([self.k_cache, K], dim=2)

# Do this (linear):
self.k_cache[:, :, self.position:self.position + seq_len] = K
self.position += seq_len
```

Each step now performs exactly $O(1)$ memory operations—just writing the new KV pair to the next slot. The cumulative cost over $N$ steps drops from $O(N^2)$ to $O(N)$.

> [!tip] The Key Insight
>
> Pre-allocate once, write to the next slot. No allocation, no copying of existing data. The only memory operation is writing the new KV pair.

```python
class MultiHeadAttentionCatCache(nn.Module):
    """KV Cache using torch.cat — allocates new memory every decode step."""

    def __init__(self, d_model: int, n_heads: int):
        super().__init__()
        self.d_model = d_model
        self.n_heads = n_heads
        self.d_h = d_model // n_heads

        self.W_q = nn.Linear(d_model, d_model, bias=False)
        self.W_k = nn.Linear(d_model, d_model, bias=False)
        self.W_v = nn.Linear(d_model, d_model, bias=False)
        self.W_o = nn.Linear(d_model, d_model, bias=False)

        self.k_cache: Optional[torch.Tensor] = None
        self.v_cache: Optional[torch.Tensor] = None

    def reset_cache(self):
        self.k_cache = None
        self.v_cache = None

    def forward(self, x: torch.Tensor) -> torch.Tensor:
        B, seq_len, _ = x.shape

        Q = self.W_q(x).view(B, seq_len, self.n_heads, self.d_h).transpose(1, 2)
        K = self.W_k(x).view(B, seq_len, self.n_heads, self.d_h).transpose(1, 2)
        V = self.W_v(x).view(B, seq_len, self.n_heads, self.d_h).transpose(1, 2)

        # INEFFICIENT: torch.cat creates new tensor and copies existing cache
        if self.k_cache is None:
            self.k_cache = K
            self.v_cache = V
        else:
            self.k_cache = torch.cat([self.k_cache, K], dim=2)  # O(N) copy!
            self.v_cache = torch.cat([self.v_cache, V], dim=2)  # O(N) copy!

        K_full = self.k_cache
        V_full = self.v_cache

        scores = torch.matmul(Q, K_full.transpose(-2, -1)) / math.sqrt(self.d_h)
        attn_weights = F.softmax(scores, dim=-1)
        output = torch.matmul(attn_weights, V_full)

        output = output.transpose(1, 2).contiguous().view(B, seq_len, self.d_model)
        return self.W_o(output)
```
```python
class MultiHeadAttentionPreallocCache(nn.Module):
    """KV Cache using pre-allocated buffers — no allocation during decode."""

    def __init__(self, d_model: int, n_heads: int, max_seq_len: int):
        super().__init__()
        self.d_model = d_model
        self.n_heads = n_heads
        self.d_h = d_model // n_heads
        self.max_seq_len = max_seq_len

        self.W_q = nn.Linear(d_model, d_model, bias=False)
        self.W_k = nn.Linear(d_model, d_model, bias=False)
        self.W_v = nn.Linear(d_model, d_model, bias=False)
        self.W_o = nn.Linear(d_model, d_model, bias=False)

        # Pre-allocated cache buffers — no allocation during decode
        self.register_buffer('k_cache', torch.zeros(1, n_heads, max_seq_len, self.d_h))
        self.register_buffer('v_cache', torch.zeros(1, n_heads, max_seq_len, self.d_h))
        self.cache_position = 0

    def reset_cache(self):
        self.k_cache.zero_()
        self.v_cache.zero_()
        self.cache_position = 0

    def forward(self, x: torch.Tensor) -> torch.Tensor:
        B, seq_len, _ = x.shape

        Q = self.W_q(x).view(B, seq_len, self.n_heads, self.d_h).transpose(1, 2)
        K = self.W_k(x).view(B, seq_len, self.n_heads, self.d_h).transpose(1, 2)
        V = self.W_v(x).view(B, seq_len, self.n_heads, self.d_h).transpose(1, 2)

        # EFFICIENT: write directly to pre-allocated buffer (no allocation)
        start = self.cache_position
        end = self.cache_position + seq_len
        self.k_cache[:, :, start:end, :] = K  # O(1) write!
        self.v_cache[:, :, start:end, :] = V  # O(1) write!
        self.cache_position = end

        K_full = self.k_cache[:, :, :end, :]
        V_full = self.v_cache[:, :, :end, :]

        scores = torch.matmul(Q, K_full.transpose(-2, -1)) / math.sqrt(self.d_h)
        attn_weights = F.softmax(scores, dim=-1)
        output = torch.matmul(attn_weights, V_full)

        output = output.transpose(1, 2).contiguous().view(B, seq_len, self.d_model)
        return self.W_o(output)
```
</CodeTabs>

The key differences:
1. **Initialization**: Pre-allocated version takes `max_seq_len` and creates buffers upfront
2. **Cache update**: Slice assignment (`cache[:, :, start:end] = K`) vs. concatenation
3. **Position tracking**: Explicit `cache_position` counter

**Trade-off**: You must know `max_seq_len` upfront. In practice, this is rarely a problem—you typically have a known context limit (4K, 8K, 128K tokens), and over-allocating by a small margin is cheap compared to the quadratic penalty.

# Caveats and Real-World Context

> [!hint] Author's Note
>
> Before you rush to refactor every KV cache implementation you see: context matters. The tutorials using `torch.cat` are optimizing for clarity, not production performance. That's the right trade-off for educational content.

A few important caveats:

**Generation length matters.** Many LLM completions are short—the model hits an EOS token or the user's requested `max_tokens` before the quadratic cost becomes significant. If your typical generation is under 1K tokens, this optimization has minimal impact.

**Long-context use cases do hit this.** Summarizing long documents, RAG pipelines with extensive retrieved context, and agentic workflows with long conversation histories all accumulate substantial context. These are exactly the scenarios where the optimization pays off.

**Production systems already know this.** If you're using a well-maintained inference framework, this is likely already handled:

- **[vLLM](https://arxiv.org/abs/2309.06180)** uses PagedAttention, which manages KV cache in fixed-size blocks—a more sophisticated form of pre-allocation that also enables memory sharing across requests.
- **[TensorRT-LLM](https://github.com/NVIDIA/TensorRT-LLM)** uses pre-allocated contiguous buffers with explicit length tracking.
- **[HuggingFace Transformers](https://huggingface.co/docs/transformers/kv_cache)** introduced `DynamicCache` (post-v4.36) which handles this more efficiently than naive concatenation.

The tutorials lag behind production practice because their goal is different: teaching the concepts, not maximizing throughput.

**What we didn't cover**: This post focused narrowly on allocation patterns. There are many other KV cache optimizations that stack with this one:

- **FlashAttention**: Fuses attention computation to reduce memory bandwidth (orthogonal to cache allocation)
- **Continuous batching**: Serves multiple requests by interleaving their decode steps
- **Quantized KV cache**: Stores keys/values in INT8 or FP8 to reduce memory footprint
- **Sliding window attention**: Limits attention to recent tokens, capping cache size

# Conclusion

A small implementation detail—`torch.cat` versus slice assignment—leads to a significant performance difference at scale. The fix is simple:

1. Pre-allocate your KV cache buffers to `max_seq_len`
2. Track the current position with an index
3. Write new KV pairs using slice assignment: `cache[:, :, pos:pos+1] = new_kv`

The broader lesson: **memory allocation patterns matter as much as algorithmic complexity**. Both `torch.cat` and slice assignment implement the same logical operation (appending to a cache), but their performance characteristics diverge dramatically as sequence length grows.

If you maintain a KV cache tutorial or implementation, consider updating it—or at least adding a note about this trade-off. And if you're deploying LLMs with long context windows, double-check that your inference stack isn't paying this hidden quadratic tax.
